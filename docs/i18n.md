# Internationalisation (i18n)

BetterTrack's UI and its notification emails are translatable. **English is the
source of truth and the default**; **German** is the first translation
(machine-seeded in V3-P1, human-corrected in V3-P13). This document is the
contract for how the layer works and, most importantly, the **add-a-language
recipe** — which is deliberately _one locale file + one registry entry, with
zero other code changes_ (PROJECTPLAN.md §13.3, V3-P1).

## Architecture

- **Locale registry — `apps/web/src/i18n/registry.ts`** is the single source of
  truth for which languages the SPA renders. It maps each locale code to a label
  (its own endonym, for the picker), an `Intl` BCP-47 locale (for number/date
  formatting), and its message catalog.
- **Message catalogs — `apps/web/src/i18n/messages/<code>.json`**, one file per
  language, nested by surface (`common`, `nav`, `auth`, `settings`, `portfolio`,
  `workboard`, `language`). `en.json` is authoritative; every other file mirrors
  its key shape. A missing key in a translation **falls back to English**, never
  a raw key.
- **Runtime — `apps/web/src/i18n/I18nProvider.tsx`**: `useT()` gives the
  `t('dot.path.key', { var })` translator (interpolates `{{var}}` tokens);
  `useI18n()` adds `{ locale, setLocale }`. The provider switches the app at
  runtime (no reload) and, on every change, drives `setFormatLocale()` and
  `<html lang>`. It has a **graceful default context** (EN + the de-AT number
  default), so a component rendered without a provider — e.g. in a focused unit
  test — still translates to the English source strings.
- **Locale-aware formatting — `apps/web/src/lib/format.ts`**: numbers, money,
  percentages and dates format via `Intl` in the active locale (`de-AT` →
  `1.234,56 €` / `15.01.2024`; `en-GB` → `1,234.56 €` / `15 Jan 2024`). Money is
  symbol-last in every locale (§7.1); dates always render in the **Europe/Vienna**
  wall-clock (§5.5).
- **Per-user persistence**: the choice lives in `users.locale` (a short BCP-47
  code; `en` by default). It is read/written via `GET`/`PATCH /settings/account`
  and rides the `/auth/me` response so the SPA seeds the runtime on load. The
  Settings → Account **language picker** switches instantly and persists, so it
  survives logout/login.
- **Localized emails**: notification emails (friend request/accepted, portfolio
  shared, price alert) render in the recipient's stored locale via
  `apps/api/src/services/email/emailI18n.ts`. This is a small, separate
  server-side catalog (a different runtime from the SPA); account/admin emails
  (invite, temp password, reset, 2FA code, welcome, SMTP test) stay English
  because their recipients often have no stored locale yet.

## Add a language (the code-free recipe)

Adding a language is **two edits and nothing else** — no component, logic, API or
schema changes. To add French:

1. **Copy the catalog and translate it.** Copy
   `apps/web/src/i18n/messages/en.json` to
   `apps/web/src/i18n/messages/fr.json` and translate the string _values_ (keep
   every key and every `{{token}}` exactly).
2. **Register it.** Add one entry to `LOCALES` in
   `apps/web/src/i18n/registry.ts`:

   ```ts
   import fr from './messages/fr.json';
   // …
   export const LOCALES = {
     en: { code: 'en', label: 'English', intlLocale: 'en-GB', messages: en as … },
     de: { code: 'de', label: 'Deutsch', intlLocale: 'de-AT', messages: de as … },
     fr: { code: 'fr', label: 'Français', intlLocale: 'fr-FR', messages: fr as … },
   } satisfies Record<string, LocaleDefinition>;
   ```

That's the whole change. The language picker, per-user persistence, `Intl`
formatting and EN fallback all pick it up automatically. **No API/DB edit is
needed**: `users.locale` and the `PATCH /settings/account` validator accept any
well-formed short code by design (they never hardcode the renderable set), and
the SPA falls back to English for any code it can't render. A translation left
partial simply falls back to English for its missing keys.

## The hardcoded-string CI gate

`pnpm lint` fails a PR that introduces a hardcoded user-facing string on a
migrated surface. The rule is `no-literal-jsx-string`
(`packages/config/rules/noLiteralJsxString.js`, unit-tested in
`noLiteralJsxString.test.js`). It flags:

- **JSX text** between tags containing a word (≥2 letters), and
- string literals in the user-facing attributes **`placeholder`, `title`,
  `alt`, `aria-label`**.

`{t('…')}` and any `{expression}` pass; `className`, `type`, `name`, `role`,
route paths, `data-*`, ids and numeric/punctuation-only text are never flagged.

### Scope and how it grows

The gate is applied by the `I18N_GATED_SURFACES` glob list in
`packages/config/eslint.config.js`. In V3-P1 it covers the fully-migrated user
surfaces: **nav** (`AppLayout`/`SubNav`/`ProfileMenu`), **`auth/**`**,
**`settings/**`**, **`portfolio/**`**, **`workboard/**`**. Every phase after
V3-P1 must route its copy through `t()` with EN + DE keys in the same PR — and
opts its surface into the gate by adding its glob to that list once extracted.

### Exemptions (documented, intentional)

Not yet gated — extracted as their owning phase migrates them:

- **Admin** (`apps/web/src/admin/**`) — admin-only internals (English by design).
- **Social, assets, OAuth** (`apps/web/src/user/social/**`, `assets/**`,
  `oauth/**`) — migrate with their V3 phases.
- **Shared widgets in `apps/web/src/user/components/**`** other than the three
nav files (e.g. `TransactionDialog`, `AlertDialog`, `CmdKPalette`,
`NotificationBell`, `AssetSearchBox`) — migrate with the surfaces that own them.
- **Shared UI** (`apps/web/src/ui/**`, `charts/**`, `lib/**`) — shared primitives
  / non-JSX.
- **Test files** (`*.test.tsx`) — they assert on the rendered EN source strings.

Two things the rule intentionally does **not** catch, handled by convention +
review: (a) user strings assembled in plain data structures / helper functions
(pass `t` into the helper — see the `assetTypeLabels(t)` / `changeErrorMessage(t)`
patterns); and (b) copy in non-`.tsx` files (e.g.
`apps/web/src/user/workboard/conglomerateBuilder.ts`).

To exempt a genuinely-technical literal on a gated surface, wrap it in an
expression container (`{'myapp://callback'}`) or, rarely, add
`// eslint-disable-next-line i18n/no-literal-jsx-string` with a one-line reason.
