how many workers make sense?
2 is the sweet spot — this repo usually has two truly independent lanes (api + web). Parallelism only exists when issues claim non-overlapping paths; workers beyond the number of open lanes just idle or serialize.
3 pays off occasionally (contracts / docs / admin lane open at the same time) at the cost of more conflict-serialization and BEHIND re-gates at the single merger.
4 is the hard cap — beyond that everything fights: one merge queue, one shared 5h token budget, overlapping touches, CI queueing. 10 would mostly be 8 idle containers burning your window.

how difficulty routing works
The composer labels every issue with exactly one of diff:easy · diff:normal · diff:intermediate · diff:hard · diff:max. Writer, fixer and ci-fix run on the model you map to the issue's difficulty; reviews run at the issue's difficulty but never below the review floor; the composer and checker master-roles use the difficulty slots above. A rejected issue escalates ONE difficulty higher on its triage retry (max stays max). Legacy tier labels still resolve: sonnet→easy, opus→intermediate, fable→max. Effort semantics: Claude = --effort (low…max) · Codex = model-supported reasoning effort (low…max/ultra) · Antigravity bakes the level into the model name, e.g. “Gemini 3.1 Pro (High)”.
